1. SOT / EOT (Start/End of Transmission)

The protocol uses SOT and EOT control packets to manage the connection lifecycle. Before any data is sent, the Sender transmits an SOT packet to the Receiver, which responds with an ACK. This establishes the session similar to a simplified handshake. If the SOT is lost (dropped by ChaosEngine), the Sender retransmits it after the timeout expires. Once all data packets have been sent and acknowledged, the Sender sends an EOT packet signaling the end of the transfer. The Receiver ACKs the EOT and writes the assembled file to disk. Both SOT and EOT are retransmitted on timeout, ensuring reliable session management even under lossy conditions.

2. Protocol Implementation

Stop-and-Wait (SAW): The Sender transmits one data packet and waits for an ACK before sending the next. If the ACK is not received within the timeout, the packet is retransmitted. This is simple but has low throughput since only one packet is in flight at a time.

Go-Back-N (GBN): The Sender can have up to N unacknowledged packets in the pipeline (window size = N). The Receiver sends cumulative ACKs for the highest in-order sequence number received. On timeout, the Sender "goes back" to the base of the window and retransmits all packets from there. Out-of-order packets are discarded by the Receiver.

4. Analysis

With no chaos (0.0), GBN is slightly faster than SAW because all packets in the window are sent at once without waiting. As chaos increases, both protocols experience more retransmissions, but GBN's impact is larger because a single lost ACK causes the entire window to be retransmitted. SAW is simpler and wastes fewer retransmissions per loss event, but its baseline throughput is lower. GBN with a larger window outperforms SAW under low-loss conditions, but the advantage shrinks as the chaos factor rises.
